# TESTING_GUIDE.md

测试与验证指南。

这份文档的目标，是把“应该怎么测”从抽象原则变成更具体、可执行的开发动作。

---

## 1. 总原则

测试的目标不是为了凑数量，而是为了：
- 防止关键 bug 回归
- 验证主链没有被拆坏
- 保证接口契约和字段语义稳定
- 让后续重构更敢继续推进

---

## 2. 最低验证线

任何非纯文档改动，至少应确认：
- build 通过
- 关键页面不报错
- 原有主流程未断

---

## 3. 按改动类型选择测试

### 3.1 解析逻辑改动
优先补：
- 解析器单测

### 3.2 状态聚合 / 字段回退 / 数值修正
优先补：
- service / controller 层测试

### 3.3 接口契约变化
必须补：
- 接口契约测试
- 至少验证关键字段语义是否仍一致

### 3.4 前端 helper / composable 改动
优先补：
- helper / composable 单测（已有测试基础时）

### 3.5 用户可见回归修复
至少补：
- 一层最贴近根因的自动化测试

---

## 4. 重构型改动怎么测

即使你认为“只是重构，行为没变”，也至少要验证：
- build 通过
- 关键页面加载正常
- 主流程没有中断
- 新旧接线没有混用

如果重构触及：
- 弹窗
- 实时展示
- 状态绑定
- 接口接线
- 页面装配层

则应额外验证最容易出回归的点。

---

## 5. 当前项目里最值得重点测的区域

- 运行中进度链
- `/api/runs/active` 相关展示
- WebSocket 更新链
- `TaskView.vue` 相关拆分点
- 弹窗开关与内容绑定
- build 与样式完整性

---

## 6. 手工验证也要有重点

手工验证不要无边界乱点。

每次改动后，应优先验证：
- 这一步直接影响的点
- 这一步最容易引发回归的点
- 这一步关联的主链是否仍自洽

例如：
- 修总量/总数显示时，重点看总量、总数、百分比是否一致
- 修 WebSocket 时，重点看不刷新问题是否消失
- 拆弹窗时，重点看开关、内容、绑定是否正常

---

## 7. 推荐测试结果表达方式

测试反馈尽量写成：
- 哪一项正常了
- 哪一项还不对
- 哪一步开始不对
- 是否比之前更好或更差

比起“有问题”“不对”，这类反馈更能帮助下一步快速定位。

---

## 8. 当前项目的常用验证动作

可按改动类型组合执行：
- 前端 build
- Go controller / service 相关测试
- 页面关键路径手工验证
- 运行中任务链专项验证

后续如测试命令和流程越来越稳定，可以再继续补成更详细的命令速查版。

---

## 9. 一个简单原则

如果一个问题已经真实出现过一次，而且以后再出现的代价很高，那它通常值得被固化进测试。
