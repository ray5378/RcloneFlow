# TESTING_GUIDE.md

测试与验证指南。

这份文档的目标，是把“应该怎么测”从抽象原则变成更具体、可执行的开发动作。

---

## 1. 总原则

测试的目标不是为了凑数量，而是为了：
- 防止关键 bug 回归
- 验证主链没有被拆坏
- 保证接口契约和字段语义稳定
- 让后续重构更敢继续推进
- 让新增/改动代码达到可合并的覆盖率门禁下限，而不是只追求总覆盖率好看

---

## 2. 最低验证线

任何非纯文档改动，至少应确认：
- build 通过
- 关键页面不报错
- 原有主流程未断

如果仓库或 CI 对新代码覆盖率有门禁要求，则还应满足：
- 新增/改动代码的覆盖率达到门禁下限后，才算可合并
- 补测试时优先命中新逻辑、新分支和新风险点，而不是只为了抬高整体 coverage

当前建议采用以下门禁基线：
- 新增/改动代码行覆盖率（new code line coverage）`>= 80%`
- 新增/改动代码分支覆盖率（new code branch coverage）`>= 70%`
- 如果当前工具链暂时只稳定提供 line coverage，则至少先卡 `new code line coverage >= 80%`
- 不用历史总体 coverage 卡当前小改动；总体 coverage 作为趋势观察值，不作为这条规则的唯一合并条件

---

## 2.1 新代码覆盖率怎么量

为了让“new code coverage”可执行，而不是口号，默认按下面口径处理：

1. 以本次变更的 diff 作为“new code”范围。
   - 优先使用 PR diff / merge request diff
   - 本地没有 PR 环境时，用当前分支相对目标分支（通常是 `master`）的 diff

2. 优先看 changed lines / changed branches 的覆盖情况。
   - 能按 diff 行统计就按 diff 行统计
   - 能按 diff 分支统计就按 diff 分支统计
   - 如果工具只支持按文件或函数统计，则用“改动文件/改动函数的新增逻辑是否被直接命中”作为近似兜底，并在结论里明确说明口径

3. 默认合并判断顺序：
   - 先看 new code coverage 是否达到门禁
   - 再看相关测试是否通过
   - 最后再看总体 coverage 是否异常下滑

4. 总体 coverage 的作用：
   - 用于观察长期趋势
   - 用于发现意外大幅回退
   - 不用于否决一个已经满足 new code gate 的小改动

5. 如果当前工具链还没有自动 diff coverage：
   - 至少在评审/提交说明里列出“本次新增或修改了哪些函数/分支”
   - 明确说明这些点分别由哪些测试命中
   - 没有命中的点要标出来，而不是默认算通过

6. 推荐执行口径：
   - Go：优先生成 `coverprofile`，结合本次 diff 关注改动函数和分支
   - Frontend：优先使用 Vitest coverage，并对照本次改动文件/逻辑核对新增分支是否被命中
   - CI 以后如果接入 diff coverage 工具，就以 CI 报告为准；本地口径作为预检查

## 3. 按改动类型选择测试

### 3.1 解析逻辑改动
优先补：
- 解析器单测

### 3.2 状态聚合 / 字段回退 / 数值修正
优先补：
- service / controller 层测试

### 3.3 接口契约变化
必须补：
- 接口契约测试
- 至少验证关键字段语义是否仍一致

### 3.4 前端 helper / composable 改动
优先补：
- helper / composable 单测（已有测试基础时）

### 3.5 用户可见回归修复
至少补：
- 一层最贴近根因的自动化测试

## 4. 重构型改动怎么测

即使你认为“只是重构，行为没变”，也至少要验证：
- build 通过
- 关键页面加载正常
- 主流程没有中断
- 新旧接线没有混用

如果重构触及：
- 弹窗
- 实时展示
- 状态绑定
- 接口接线
- 页面装配层

则应额外验证最容易出回归的点。

---

## 5. 当前项目里最值得重点测的区域

- 运行中进度链
- `/api/runs/active` 相关展示
- WebSocket 更新链
- `TaskView.vue` 相关拆分点
- 弹窗开关与内容绑定
- build 与样式完整性

---

## 6. 手工验证也要有重点

手工验证不要无边界乱点。

每次改动后，应优先验证：
- 这一步直接影响的点
- 这一步最容易引发回归的点
- 这一步关联的主链是否仍自洽

例如：
- 修总量/总数显示时，重点看总量、总数、百分比是否一致
- 修 WebSocket 时，重点看不刷新问题是否消失
- 拆弹窗时，重点看开关、内容、绑定是否正常

---

## 7. 推荐测试结果表达方式

测试反馈尽量写成：
- 哪一项正常了
- 哪一项还不对
- 哪一步开始不对
- 是否比之前更好或更差

比起“有问题”“不对”，这类反馈更能帮助下一步快速定位。

---

## 8. 当前项目的常用验证动作

可按改动类型组合执行：
- 前端 build
- Go controller / service 相关测试
- 页面关键路径手工验证
- 运行中任务链专项验证

后续如测试命令和流程越来越稳定，可以再继续补成更详细的命令速查版。

---

## 9. 一个简单原则

如果一个问题已经真实出现过一次，而且以后再出现的代价很高，那它通常值得被固化进测试。

另一个必须记住的原则是：
- 做测试不只是为了“有测试”
- 新代码覆盖率如果还没达到门禁要求的下限，就不应视为可合并状态
- 默认按 `new code line coverage >= 80%`、`new code branch coverage >= 70%` 执行
- 当时间有限时，优先覆盖这次新增代码最关键、最脆弱、最容易回归的分支
