# KNOWN_PITFALLS.md

已知坑点速查表。

这份文档用于记录那些已经在项目中反复出现、容易误判、容易再次踩到的坑。

---

## 1. 完成态字段容易被误当运行中真源

现象：
- 运行中展示看起来能显示，但语义逐渐偏离实时主链

提醒：
- 当前运行中 UI 主真源应是 `/api/runs/active.progress`
- 应明确区分：`progress`（运行中）、`finalSummary`（历史详情）

---

## 2. `preflight` 容易被误当运行中总量真源

现象：
- 运行中总量、ETA、百分比混入预估语义

提醒：
- `preflight` 主要用于预估展示
- 不应直接驱动运行中主展示

---

## 3. 前端 masking 很容易看起来“修好了”但根因没修

现象：
- 前端显示变正常了，但后端数据链仍然不自洽

提醒：
- masking 可以临时止血
- 不能长期替代源头修复

---

## 4. `TaskView.vue` 编辑时容易留下尾部脏片

现象：
- 重复 `</style>`
- 模板 / 样式残片
- build 报 `Invalid end tag`

提醒：
- 大文件频繁精确编辑后，要优先检查尾部结构

---

## 5. WebSocket 更新容易按错字段匹配

现象：
- 后端在推送，但前端卡片不刷新

提醒：
- `run_progress` 更新 active run 时，应按 `runRecord.id` 匹配

---

## 6. `map[string]any` 容易让字段语义漂移

现象：
- 同一个字段看起来到处都能读，但没人说得清它到底代表什么

提醒：
- 核心契约和关键状态应尽量 typed struct 化

---

## 7. 构建产物容易污染工作区

现象：
- `web/index.html`
- `web/assets/*`

经常在 build 后变脏，干扰判断真实源码变更。

提醒：
- 提交前单独检查构建产物是否真的属于本轮改动

---

## 8. 把缓存当第一嫌疑对象，容易浪费时间

现象：
- 用户已经明确本地无缓存构建，但还优先怀疑旧 bundle / 旧缓存

提醒：
- 在明确 `--no-cache` 构建前提下，应先检查当前源码与接线

---

## 9. 大范围重构不分步，回归很难定位

现象：
- 一次改太多，出问题后很难知道是哪个环节引起的

提醒：
- 高风险区域优先小步拆分、每步停测

---

## 10. `TaskView.vue` 这类大装配文件容易出现隐式依赖顺序问题

现象：
- 本地看起来只是“整理顺序 / 提升可读性”
- 生产压缩后却出现 `Cannot access 'Xx' before initialization`
- 页面空白，但源码表面上看不到明显语法错误

提醒：
- 顶层 composable / handler / bridge 之间如果存在真实依赖，必须按依赖顺序显式组织
- 典型危险信号：
  - A composable 依赖 B 暴露出的 ref / handler
  - 顶层 helper 依赖另一个 composable 返回的函数
  - 为了“排版更漂亮”而重排声明顺序
- 对这类文件，不能靠“换个顺序先跑起来”当作修复完成；应把依赖顺序写实，并在验证通过后清掉临时施工脚本
