# ENGINEERING_RULES.md

开发约束与技术要求。用于约束后续开发行为，避免把当前已经理顺的链路重新改乱。

## 0. 总原则：大功能必须合理拆分成小功能

大功能不应长期堆积在单一页面、单一组件或单一模块里。凡是已经形成独立职责、独立交互、独立状态链、独立展示区块的能力，都应优先拆分成更小、更清晰、更可维护的单元。

### 0.1 目标
- 让每个模块职责单一、边界清晰
- 降低改动时的连带影响
- 提高可维护性、可复用性、可测试性
- 避免出现“文件过大、状态过多、没人敢动”的局面

### 0.2 必须遵守的拆分方向
- 大功能 -> 合理拆分成多个小功能、小组件、小模块
- 大页面 -> 拆成多个小组件
- 大交互块 -> 拆成独立组件或子模块
- 可复用逻辑 -> 拆成 helper / composable / service
- 调试逻辑 -> 与业务展示逻辑分离
- 展示逻辑、状态管理、网络更新逻辑 -> 尽量不要长期混在同一文件中

### 0.3 明确原则
- 大功能应该合理拆分成小功能，以提升可维护性、可测试性、可复用性
- 不拆分组件、长期把多个职责堆进单个 view 文件，会直接导致文件过大、状态过多、维护困难，这种做法不应继续重复

### 0.4 危险信号
出现以下情况时，应优先考虑拆分，而不是继续堆代码：
- 单文件同时负责展示、状态管理、网络更新、弹窗、调试信息
- 新功能只能通过继续追加局部状态、临时函数、模板分支来完成
- 同一个文件开始承载多个本可独立维护的 UI 区块
- 修改一个功能时，必须理解整页的大量无关逻辑

### 0.5 禁止事项
- 不要为了“先做完”而把大功能继续直接塞进现有超大文件
- 不要把本应独立的功能长期挂靠在父级 view 中不拆
- 不要依赖文件继续膨胀来换取短期开发速度

## 1. 分支规则（必须遵守）

- `dev`：开发分支
- `master`：主线 / 稳定分支
- 正确流程：`dev` 开发 -> 测试验证 -> `dev -> master`
- 不再使用 `next-master` 作为主线
- 不允许从 `master` 直接开发
- 不允许执行 `master -> dev` 的反向合并

## 2. 修改范围与提交规则（必须遵守）

- 一次改动应尽量聚焦一个目标：拆分、修 bug、清理、文档、测试，尽量不要无边界混杂
- 如果一次提交同时包含“重构 + 行为修复”，提交说明中必须写清楚两者边界
- 不要在同一轮里无节制扩大修改面；一旦开始出现回归，应先停下来修稳，再继续后续拆分
- 拆分型改动优先保持行为等价；如行为发生改变，必须明确说明改的是结构还是行为

## 3. 运行中进度源规则（必须遵守）

### 2.1 运行中 UI 主数据源
运行中 UI 必须优先使用：
- `/api/runs/active.progress`

包括但不限于：
- 任务卡片
- 运行中提示小窗
- 运行中 ETA
- 运行中进度文案

### 2.2 字段语义
- `progress`：运行中的实时进度（live frame）
- `stableProgress`：兼容字段 / 完成态固化，不得再作为运行中 UI 主数据源
- `preflight`：预估总量（来自 `rclone size` + 过滤条件），仅用于预估展示

### 2.3 禁止事项
- 不要再让运行中 UI 优先读取 `stableProgress`
- 不要再让运行中 ETA 回退到 `preflight`
- 不要在前端对运行中进度做“历史最大值拼接”或二次抗噪合并
- 不要把 `preflight` 当作运行中总量真源

## 3. 运行中 ETA 规则（必须遵守）

运行中 ETA 只允许基于 live progress 计算：
- `live.totalBytes`
- `live.bytes`
- `live.speed`

如果 `live.totalBytes` 缺失：
- 直接不显示 ETA / 返回空
- 不再回退 `preflight.totalBytes`

## 4. 日志解析规则（必须遵守）

### 4.1 aggregate progress 解析规则
聚合进度解析只允许接受完整的 aggregate one-line 统计行。

以下内容不得再被当成总进度行：
- 文件级进度行
- `Copied (new)`
- `Deleted`
- 其他不完整的碎片日志

### 4.2 size pair 解析规则
aggregate size pair 必须要求显式字节单位（例如 `MiB/GiB`）。

禁止再次允许这种宽松匹配：
- 把时间戳里的 `2026/04` 误解析成 `bytes/totalBytes`

## 5. Active Runs 接口规则（必须遵守）

`/api/runs/active` 当前是运行中 UI 的唯一事实来源。

接口约定：
- `progress`：运行中实时进度主字段
- `stableProgress`：兼容字段 / 完成态固化
- `progressLine`：最后成功解析到的原始进度日志行
- `progressSource`：当前进度来源
- `progressMismatch` / `progressCheck`：后端一致性自检结果

后续开发不得：
- 删除 `progress` 并要求前端重新切回 `stableProgress`
- 让 `stableProgress` 再次承担运行中主展示职责

## 6. WebSocket / 前端刷新规则（必须遵守）

- `run_progress` 更新 active run 时，必须按 `runRecord.id` 匹配
- 不要再按错误顶层字段匹配（例如 `r.id`）
- 运行中刷新允许存在轻量轮询兜底，但 WebSocket 仍是首选实时来源

## 7. 预检（preflight）规则（必须遵守）

### 7.1 允许用途
`preflight` 只允许用于：
- 预估总量展示
- 预估文件数展示
- 缺失时作为极小范围的兼容兜底（仅在后端链路明确需要时）

### 7.2 不允许用途
`preflight` 不应再直接驱动：
- 运行中任务卡片主进度
- 运行中 ETA
- 运行中小窗主展示

## 8. 前端组件规则（建议强约束）

### 8.1 禁止继续把大块功能堆进单文件视图
`frontend/src/views/TaskView.vue` 过大的根本原因，是长期把多个职责直接堆进一个页面文件，而没有及时拆成组件或 composable。

后续必须避免重复这个模式：
- 不要继续把任务列表、历史记录、运行中提示、弹窗、进度格式化、WebSocket 更新、调试逻辑继续直接堆进同一个 view 文件
- 新增功能如果已经形成独立 UI 区块或独立状态链，应优先拆分为组件或 composable
- 当某个 view 同时承担“展示 + 状态管理 + 网络更新 + 调试逻辑”时，应视为需要拆分的危险信号

### 8.2 拆分优先原则
新增或重构前端逻辑时，优先按以下边界拆分：
- 独立弹窗 -> 独立组件
- 独立卡片区域 -> 独立组件
- 可复用的进度/格式化/调试逻辑 -> helper 或 composable
- 与主页面弱耦合的交互块 -> 子组件

### 8.3 明确禁止事项
- 不要为了“先快点做完”而把新功能直接追加进 `TaskView.vue` 尾部
- 不要通过继续增加局部状态、临时函数、模板条件分支来维持超大页面文件
- 不要把本该在子组件中的 hover/弹窗/调试逻辑长期滞留在父级 view 中

### 8.4 其他前端约束
- 新增运行中相关展示时，优先考虑独立组件或 helper/composable
- 视觉相似的卡片，不要依赖偶然命中的全局类名去复用 hover 效果

## 9. 构建产物规则（建议强约束）

- 前端构建后的 `web/index.html` 与 `web/assets/*` 容易污染工作区
- 修改源码后，应避免把构建产物误当成业务代码改动提交
- 分支切换后若出现构建产物脏状态，应先清理再继续开发

## 10. 单一数据真源原则（必须遵守）

- 新功能开发前，必须先明确主数据源（single source of truth）
- UI 不允许长期并行依赖多套语义相近但来源不同的数据，除非明确写出回退顺序和原因
- 一旦某条数据链已确定为主源，后续展示逻辑应优先围绕该主源展开，而不是继续在 UI 层混合拼接其他来源

## 11. 回退逻辑规则（必须遵守）

- 所有回退逻辑必须带语义注释，说明：
  - 为什么要回退
  - 回退仅用于什么场景
  - 是否属于兼容逻辑
- 没有注释的回退逻辑，默认视为待清理技术债
- 回退逻辑不得替代主链本身的治理
- 不允许为了“先显示出来”而长期保留无说明的多级回退

## 12. 调试能力规则（必须遵守）

- 调试信息必须可开关
- 默认用户界面中不应长期裸露调试详情
- 原始日志、接口 JSON、自检结果等调试信息，应与主业务展示分离
- 调试能力应服务于定位问题，而不是长期混入常规展示路径

## 13. 测试补充规则（必须遵守）

- 核心 bug 修复后，必须补至少一层测试
- 尤其是以下链路：
  - 解析器
  - 状态聚合
  - 接口契约
  - 关键格式转换
- 不允许只加判断、不补测试就结束核心链路修复

## 14. 临时方案规则（必须遵守）

- 所有临时方案都必须写清楚：
  - 解决什么问题
  - 作用范围
  - 后续退出条件
- 没有退出条件的临时方案，默认按技术债处理
- 不允许把临时兼容、临时回退、临时兜底长期伪装成正式方案

## 15. Build 质量规则（必须遵守）

- 新增代码不允许引入新的 build warning
- 已知 warning 应逐步清零，不应长期积压
- 构建 warning 不能被当作常态噪音
- 一旦 build 输出中有新 warning，应优先确认是否为新增代码引入

## 16. 本地构建事实规则（必须遵守）

当用户已明确说明：
- 使用本地工作目录 `/root/.openclaw/workspace/rcloneflow/` 中的代码构建
- 并使用 `docker build --no-cache ...` 或等价方式进行无缓存构建

则默认应视为：
- 用户构建的是当前本地最新代码
- 运行结果应优先按“当前源码问题”排查
- 不应先默认归因为浏览器缓存、旧 bundle、旧镜像或旧产物，除非已有明确证据

排查这类问题时应优先检查：
- 当前源码是否接线完整
- 当前模板/脚本是否存在残留旧引用
- 当前构建产物是否与最新源码一致

不要在缺乏证据时，把问题先解释为“你本地没更新到新 bundle”。

## 17. 源头修复优先规则（必须遵守）

- 当问题已明确可追溯到后端数据、接口组装、解析链、状态聚合链时，应优先修源头，而不是先在前端做遮盖式修补
- 前端展示层可以临时止血，但不得把前端 masking 当作最终修复方案
- 如果最终采用临时前端止血方案，必须记录退出条件，并在源头修复完成后尽快移除
- 当用户明确要求“查根因 / 修源头”时，不应继续默认走前端补丁路线

## 18. 接口契约与文档同步规则（必须遵守）

- 已被多个前端点位依赖的接口字段，一旦形成事实标准，必须在文档或测试中体现其契约
- 修改接口字段含义、优先级、回退条件时，必须同时更新：
  - 测试
  - 开发文档
  - 相关注释（如有）
- 不允许只改行为、不补契约说明

## 19. 命名与语义一致性规则（必须遵守）

- 同一概念在同一层内应尽量使用同一名称，不要同时混用多个近义变量名
- 名称必须尽量体现真实语义，例如：
  - 预估值应带 `preflight` / `estimated`
  - 实时值应带 `progress` / `live`
  - 完成态应带 `final` / `stable`
- 不要用会掩盖语义差异的通用名称，让后续维护者误以为不同来源是同一个概念

## 20. 双轨实现与残留代码清理规则（必须遵守）

- 重构完成后，旧接线、旧状态、旧模板引用、旧 helper 如不再使用，必须及时清理
- 不允许长期保留“新旧两套并行但只有一套在用”的双轨结构
- 若短期内必须并存，必须明确标注哪套是主链、哪套是兼容层，以及计划移除条件
- 出现页面空白、重复弹窗、变量未定义、残留旧引用等问题时，应优先排查是否存在双轨残留

## 21. 回归处理规则（必须遵守）

- 一旦拆分或重构引发用户可见回归，应先修复当前回归，不应继续推进后续拆分
- 回归修复完成后，应总结根因，并在必要时补文档或补测试
- 对用户可直接验证的改动，应在继续下一刀前明确告知影响面和测试点
- 对交互层、显示层、状态绑定层的改动，默认应采用“小步修改 -> 用户验证 -> 再继续”的节奏

## 22. 提交前自检规则（必须遵守）

提交前至少确认：
- 目标文件没有残留旧引用、死代码、尾部脏片、重复块
- 相关 build / test 已通过
- 改动范围与提交说明一致
- 没有把无关构建产物、无关样式噪音、无关调试代码一起提交
- 若用户将直接基于本地工作区构建，应默认以当前源码状态为准，不应留“本地一跑就炸”的接线缺口

## 23. 遇到运行中进度问题时的排查顺序（必须遵守）

排查顺序固定为：

`日志原文 -> summary.progress -> /api/runs/active.progress -> 任务卡片 / 运行中提示小窗`

不要直接从以下字段逆推运行中 UI：
- `preflight`
- `stableProgress`

除非你明确是在处理：
- 完成态固化
- 历史兼容逻辑
- 缺失字段兜底
