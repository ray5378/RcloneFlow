# BACKEND_RULES.md

后端开发规则。用于约束 Go / HTTP 接口 / 状态聚合 / 执行链代码在本项目中的实现方式，减少字段语义漂移、接口契约混乱和定位困难的问题。

## 1. 总目标

后端代码应优先追求：
- 契约清楚
- 语义稳定
- 分层明确
- 错误可定位
- 关键链路可测试

不要只追求“先把 JSON 拼出来”，而忽视字段语义、边界和后续维护成本。

## 2. 分层职责规则

### 2.1 controller
controller 负责：
- HTTP 入参解析
- 出参组织
- 状态码返回
- 轻量接口聚合

controller 不应长期承担：
- 大量底层解析细节
- 无边界业务规则
- 难以复用的大段流程编排

### 2.2 service
service 负责：
- 业务规则
- 状态聚合
- 多来源数据的协调
- 字段优先级、兼容策略、语义修正

service 应尽量成为“为什么这样返回”的主要承载层。

### 2.3 store / adapter / runner
- `store`：持久化与数据读取
- `adapter`：外部系统适配
- `runner`：执行链与运行态处理

这些层不应反向掺入页面语义，也不应为了前端显示方便直接携带展示层拼接逻辑。

## 3. 接口契约规则

### 3.1 字段语义必须稳定
一旦某个接口字段被多个前端点位依赖，就应视为事实契约。

不得随意改变：
- 字段含义
- 优先级
- 回退顺序
- 零值语义

### 3.2 契约变更要求
如果接口契约发生变化，必须同步：
- 测试
- 文档
- 必要注释

### 3.3 字段命名
- 名称应体现真实语义，不要用模糊名称掩盖 live / stable / preflight / final 的区别
- 不同语义的字段不要用同一名字在不同位置反复重载

## 4. 类型与数据结构规则

### 4.1 typed struct 优先
- 能用明确 struct 表达的数据，不要长期停留在 `map[string]any`
- 核心接口输出、关键状态对象、固定字段集合，应优先 typed struct

### 4.2 `map[string]any` 使用限制
以下场景可短期使用 `map[string]any`：
- 外部不稳定输入的过渡解析
- 历史数据兼容读取
- 临时调试或探测性字段

但不得长期把 `map[string]any` 当成主要业务建模手段，尤其是：
- 核心接口返回
- 关键状态聚合对象
- 多处复用的数据结构

### 4.3 零值与缺失值
- 必须明确区分：字段不存在、字段为零值、字段语义未知
- 不要默认把零值当作“正常且可信的业务值”
- 需要回退时，应明确说明回退条件

## 5. 错误处理规则

### 5.1 错误必须可解释
- 返回错误时，应尽量保留足够语义，便于定位问题来源
- 不要大量返回含义模糊的通用错误文本

### 5.2 不要随意吞错
- 吞错必须有明确理由，例如兼容读取、可选字段、探测性逻辑
- 吞错不应影响主链正确性判断
- 如果错误被吞掉，必要时应有日志或注释说明

### 5.3 controller 错误边界
- controller 应负责把错误转成合理的 HTTP 响应
- 不要在 controller 层堆大量与业务耦合的错误修补逻辑

## 6. 状态聚合与回退规则

### 6.1 单一真源优先
- 状态聚合前，先明确主数据源
- 回退逻辑应围绕主链缺失或兼容场景服务，而不是与主链长期并行竞争

### 6.2 回退逻辑必须可解释
- 写回退时要能回答：
  - 为什么回退
  - 在什么条件下回退
  - 回退值的可信度是什么
- 不要为了“先别空着”就加多层兜底，把语义搅乱

### 6.3 字段自洽
- 同一响应里的相关字段应尽量保持一致
- 不允许出现：总量来自一套语义、百分比来自另一套语义、计数来自第三套语义，却没人知道谁才可信

## 7. JSON 组装规则

- JSON 输出应优先在靠近契约层的位置统一组装
- 不要让多个 controller/调用点各自组装同一套字段语义
- 关键字段的默认值、裁剪规则、修正规则应集中表达，避免散落

## 8. 日志与可观测性规则

### 8.1 日志目标
日志主要用于：
- 定位错误
- 还原状态变化
- 判断真实数据链是否正确

### 8.2 日志要求
- 关键链路上的日志应能说明阶段、来源、关键字段或关键决定
- 不要只记录“进入了这里”“执行了那里”这种无信息量日志
- 不要让调试日志与正式契约混为一体

## 9. context 与生命周期规则

- 需要响应请求生命周期或取消信号的操作，应正确传递 `context`
- 不要无理由忽略 `context`
- 长流程、外部调用、可取消任务，应优先围绕 `context` 组织

## 10. 幂等与重复调用意识

- 对可能被重复调用的接口、状态更新、停止/启动类动作，应提前考虑重复调用结果
- 不要默认调用只会发生一次
- 同一操作多次执行时，结果应尽量稳定且可解释

## 11. Go 测试规则

### 11.1 表驱动优先
- 对解析器、状态修正、字段优先级、边界条件，优先使用表驱动测试

### 11.2 测试层次
- 解析逻辑 -> 单元测试
- service 规则 -> service 测试
- controller JSON / HTTP 输出 -> handler / controller 测试
- 契约变更 -> 接口契约测试

### 11.3 回归测试
- 已定位过的关键 bug，应尽量补成可长期防回归的测试
- 不要只在聊天里说“以后注意”，却不固化到测试

## 12. 后端高风险区提醒

以下区域改动时默认高风险，应优先补测试并明确契约影响：
- 日志解析
- 状态聚合
- 回退优先级
- 接口字段语义
- 实时进度链
- 兼容旧数据读取

## 13. 一个简单判断标准

如果一个后端字段的值来源，必须一路追：
- 外部输入
- 历史兼容
- 多层回退
- 若干 map 字段
- controller 临时修补

才能知道它到底代表什么，那通常说明这条链已经需要收敛和重构了。
