# DEVELOPMENT_CHECKLIST.md

日常开发执行清单。

这份文档不是用来替代 `ENGINEERING_RULES.md`，而是把最常用、最需要在开发过程中反复执行的规则压缩成一份短清单，方便在真正改代码时快速对照。

---

## 1. 开发前检查

开始改代码前，先确认以下几点：

- [ ] 当前分支是否正确：开发默认在 `dev`
- [ ] 这次改动的目标是否单一、清楚
- [ ] 这次改动属于哪一类：
  - [ ] 修 bug
  - [ ] 拆分 / 重构
  - [ ] 文档
  - [ ] 测试
  - [ ] 样式
- [ ] 主数据源（single source of truth）是否已经明确
- [ ] 是否已经知道该逻辑应该放在哪一层：
  - [ ] view
  - [ ] component
  - [ ] composable
  - [ ] helper
  - [ ] api / controller / service
- [ ] 如果是运行中进度问题，是否按固定排查顺序开始：
  - `日志原文 -> summary.progress -> /api/runs/active.progress -> 前端展示`

---

## 2. 改动时检查

写代码过程中，持续检查：

- [ ] 没有无边界扩大修改范围
- [ ] 没把多个目标硬塞进同一轮改动
- [ ] 没把本应独立的 UI 区块继续堆进超大 view 文件
- [ ] 没把纯格式化 / 纯转换逻辑继续塞在页面主文件里
- [ ] 没在多个前端点位各自做后端字段兜底
- [ ] 没把临时止血方案当成最终方案
- [ ] 修回归时，没有把问题藏进新的隐式依赖、顺序耦合、临时脚手架或未清理补丁里
- [ ] 如果问题能定位到后端链路，优先修源头
- [ ] 没新增新的双轨实现 / 新旧两套并行接线
- [ ] 调试信息仍然可开关，没有混进默认展示路径

如果改动触及以下区域，要默认采用“小步修改 -> 测试/验证 -> 再继续”：

- [ ] 交互层
- [ ] 显示层
- [ ] 状态绑定层
- [ ] 弹窗层
- [ ] 实时数据链
- [ ] 接口组装链
- [ ] 日志解析链

---

## 3. 拆分 / 重构专项检查

如果这次改动属于拆分或重构，额外确认：

- [ ] 这次主要是在改结构，不是在偷偷改行为
- [ ] 如果行为确实有变化，已经明确记录影响面
- [ ] 已拆出的逻辑边界清楚，放置位置合理
- [ ] 旧接线、旧状态、旧模板引用、旧 helper 已同步清理
- [ ] 新增 composable 后，是否已立即核对：import 已补齐、页面解构已接回、页面旧状态是否删干净、调用点是否都已切到新来源
- [ ] 没留下页面空白、变量未定义、重复弹窗、旧引用残留这类典型双轨问题
- [ ] 出现回归时，先修稳当前回归，而不是继续往下拆
- [ ] 当前这一步是否已经真正达到值得用户测试的门槛；如果还只是接线、占位、局部派生迁移，则继续推进，不要过早打断用户
- [ ] 如果已经达到测试点，是否已先执行 `docker build --no-cache -t ray5378/rcloneflow:latest .`
- [ ] 如果当前改动属于修错误 / 修回归 / 修运行时异常 / 修页面空白 / 修接线错误，即使改动很小，是否也已按测试点规则重新执行 no-cache 镜像构建，而不是只停留在本地 build 通过
- [ ] no-cache 构建开始后，是否未再全程盯看，而是统一到约 5 分钟时再检查一次结果
- [ ] 5 分钟检查时，无论构建结果是成功、失败还是仍未完成，是否都已直接把结果告知用户
- [ ] 该 5 分钟检查与结果通知，是否由 Agent 主动触发，而不是等用户追问后才处理
- [ ] 是否未再自动重试 no-cache 构建

---

## 4. 测试 / 验证检查

改完后至少确认：

- [ ] 前端 build 已通过
- [ ] 关键页面不报错
- [ ] 原有主流程未断

如果改动属于以下情况，还需要对应补测试：

### 4.1 解析逻辑变更
- [ ] 补解析器单测

### 4.2 状态聚合 / 字段回退 / 数值修正变更
- [ ] 补 service / controller 层测试

### 4.3 接口字段语义、优先级、回退规则变更
- [ ] 补接口契约测试

### 4.4 前端 helper / composable 改动
- [ ] 在已有测试基础上补对应单测

### 4.5 用户可见回归修复
- [ ] 至少补一层最贴近根因的自动化测试

---

## 4.6 构建依赖缓存检查

如果当前改动涉及 Docker 构建耗时优化，额外确认：

- [ ] 是否优先选择了“仓库内本地缓存 + 联网回退”而不是纯宿主机绝对路径依赖
- [ ] 是否只对版本固定、体积大、收益明显的依赖启用该策略
- [ ] 是否保留了缓存缺失时的远端 fallback
- [ ] 是否避免把高波动依赖整体塞进仓库（如 npm 全量依赖、Go 全量 vendor、apk 包缓存）
- [ ] 如果问题出在 Docker 基础镜像拉取（`FROM node:...` / `FROM golang:...`），是否优先使用预加载脚本（如 `scripts/docker/load-base-images.sh`）而不是把基础镜像下载逻辑硬塞进 Dockerfile

## 5. 提交前检查

提交前逐项确认：

- [ ] 目标文件没有尾部脏片、重复块、残留旧引用
- [ ] 没有死代码、无用状态、废弃接线残留
- [ ] build / test 已通过
- [ ] 提交范围与提交说明一致
- [ ] 若本轮改动触发了前端重新打包，已同步检查并提交 `web/index.html` 与 `web/assets/*`
- [ ] 没把与本轮无关的旧构建产物噪音一起提交
- [ ] 没把无关样式噪音、无关调试代码一起提交
- [ ] 如果本轮新增或修改了默认配置项，已确认：默认值、保存、重开回显、保存后立即生效
- [ ] 若用户将直接基于本地源码构建，当前源码状态可直接工作

### 5.1 提交信息检查
- [ ] 提交信息不是模糊词（如 `fix` / `update` / `调整一下`）
- [ ] 推荐格式：`type(scope): summary`
- [ ] type 使用合理：`fix` / `refactor` / `docs` / `test` / `style` / `chore`
- [ ] scope 能帮助定位真实模块或链路
- [ ] summary 能描述“改完之后的结果”

## 6. 出回归时的处理顺序

一旦出现用户可见回归，不要硬顶着继续做后续功能。按这个顺序处理：

- [ ] 先停下当前扩展改动
- [ ] 先确认回归影响面
- [ ] 先修复当前回归
- [ ] 必要时补测试 / 补文档
- [ ] 再决定是否继续下一刀

如果回归涉及：
- 页面空白
- 强制弹窗
- 变量未定义
- 内容全空
- 运行中数据显示自相矛盾

优先排查：
- [ ] 新旧接线是否混用
- [ ] 旧引用是否残留
- [ ] 主数据源是否被旁路
- [ ] 前端是否做了不该长期存在的 masking

---

## 7. 文档更新检查

以下情况，改动完成后应主动判断是否补文档：

- [ ] 新增或修订了开发规则
- [ ] 接口字段语义、优先级、回退条件发生变化
- [ ] 主链路、主真源、排查顺序发生变化
- [ ] 某项技术债被确认 / 被消除
- [ ] 某个超大文件拆分已形成阶段性进度
- [ ] 某次回归修复形成了值得长期保留的经验

### 7.1 文档落点判断
- 长期工程规则 -> `docs/ENGINEERING_RULES.md`
- 技术债、拆分进度、遗留问题 -> `docs/TECH_DEBT.md`
- 日常执行版检查清单 -> `docs/DEVELOPMENT_CHECKLIST.md`

---

## 8. 运行中进度链专项提醒

只要碰运行中显示、ETA、总量、总数、小窗、任务卡片，就优先复查这几条：

- [ ] 运行中 UI 主数据源仍是 `/api/runs/active.progress`
- [ ] 没把任务卡片结束态过渡重新拉回运行中主展示字段
- [ ] 没把带 `Stable` 语义的 helper / runtime 命名重新引回运行中主链
- [ ] 没把 `preflight` 重新当成运行中真源
- [ ] ETA 仍只基于 live progress 算
- [ ] 没在前端重新加“历史最大值拼接”式抗噪逻辑
- [ ] 问题排查仍按固定顺序进行

## 9. 一个简单原则

如果你做完这次改动之后，后续维护者更难理解、更难定位、更难验证，通常说明这次改动方式有问题。

目标不是“先跑起来”，而是：

- 更清楚
- 更稳定
- 更容易继续改
- 更不容易再把同一类问题改回来
