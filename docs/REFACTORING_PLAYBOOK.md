# REFACTORING_PLAYBOOK.md

低风险重构与拆分手册。

这份文档的目标，是把“怎么拆、怎么重构、怎么在不把系统拆坏的前提下持续推进”沉淀成可执行方法，而不是只停留在原则层。

---

## 1. 总原则

重构的目标不是“看起来更漂亮”，而是：
- 让结构更清楚
- 让职责边界更明确
- 让后续改动更容易
- 让回归风险更低

如果一次重构让系统更难理解、更难定位、更难验证，那通常不是成功的重构。

---

## 2. 推荐的重构顺序

对于超大页面、复杂状态链或高风险链路，推荐顺序：

1. 先识别主链与真源
2. 先固定不动的部分
3. 先拆纯 helper / 纯格式化逻辑
4. 再拆 composable / 可复用状态逻辑
5. 再拆独立 UI 组件
6. 最后再处理页面级编排层收口

不要一上来就大范围搬模板和状态。

---

## 3. 小步拆分规则

### 3.1 一次只做一刀
每次只拆一个明确目标，例如：
- 拆一个弹窗
- 拆一个 helper
- 拆一个 composable
- 拆一个列表区域

### 3.2 每刀都要可验证
每次改动后，应能明确回答：
- 这一步只改了什么
- 这一步可能影响什么
- 这一步改完要重点测什么

### 3.3 不要连续无停顿推进多刀
高风险重构不适合“连续做完一大串再统一测”。

更稳的方式是：
- 改一刀
- 验一刀
- 稳一刀
- 再继续下一刀

---

## 4. 哪些内容优先拆

优先拆的通常是：
- 独立弹窗
- 独立卡片区域
- 纯格式化逻辑
- 纯调试逻辑
- 可复用状态读取逻辑
- 与主页面模板弱耦合的交互块

后拆的通常是：
- 页面级全局协调逻辑
- 多区域共享的复杂状态编排
- 牵涉多个链路的主流程

---

## 5. 拆分过程中最容易出的问题

典型回归包括：
- 漏导入
- 旧引用残留
- 新旧接线混用
- 弹窗状态脱节
- 页面空白
- build 失败
- 文件尾部脏片
- 局部功能看似正常，但数据链语义已悄悄变掉

因此每次拆分后都要优先检查：
- 接线是否完整
- 旧逻辑是否已清理
- 主数据源是否仍一致
- build 是否通过

---

## 6. 什么时候应先停下，不继续拆

出现以下情况时，应优先停下修稳：
- 页面空白
- 关键交互失效
- 弹窗异常
- 数据展示明显错乱
- 控制台直接报错
- 主链字段语义开始混乱

不要一边带着回归，一边继续往下拆。

---

## 7. 行为不变与行为改变要分清

### 7.1 结构重构
如果目标是纯结构优化，应尽量保持行为等价。

### 7.2 行为修复
如果在重构中顺手修了 bug，必须明确说清：
- 哪部分是结构变化
- 哪部分是行为变化

不要把“修 bug”伪装成“只是重构”。

---

## 8. 重构后的收口动作

重构完成后至少要做：
- build 验证
- 关键页面验证
- 旧代码清理
- 必要测试补充
- 文档更新

对于阶段性重构，还应补：
- 当前拆分进度
- 已完成模块
- 剩余待拆范围

---

## 9. 当前项目中已验证有效的方法

以下方法已经在当前项目中被证明有效：
- 小步拆分
- 每步停测
- 先拆 helper / composable，再拆 UI
- 出回归先修稳
- 对实时展示链路先查根因，再决定前后端怎么改
- 不用前端 masking 伪装长期正确性

---

## 10. 一个简单原则

如果一段代码你不敢拆，是因为它同时承担了太多职责；
如果一次重构你不敢停，是因为你改得太大了。
