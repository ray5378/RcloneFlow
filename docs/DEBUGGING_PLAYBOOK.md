# DEBUGGING_PLAYBOOK.md

项目排障手册。

这份文档不是替代日志和测试，而是沉淀“遇到典型问题时，应该优先怎么查”的经验，减少重复走弯路。

---

## 1. 页面空白 / 页面直接报错

### 典型现象
- 页面加载后内容大面积空白
- 控制台直接出现 `ReferenceError` / `TypeError`
- 某次拆分后主页面无法正常显示

### 优先排查
1. 是否有漏导入
2. 是否存在旧引用残留
3. 是否新旧接线混用
4. 是否拆分后还保留了旧状态/旧模板分支
5. 是否有文件尾部脏片或模板结构残留

### 常见误区
- 先怀疑浏览器缓存
- 先怀疑旧 bundle
- 不先看当前源码接线是否完整

---

## 2. 弹窗强制出现 / 弹窗内容异常

### 典型现象
- 页面一打开就被弹窗强制覆盖
- 弹窗无法关闭
- 弹窗内容为空或全是 `-`

### 优先排查
1. 弹窗 open/visible 状态绑定是否正确
2. 当前 run / 当前 task 的绑定是否丢失
3. composable 与页面是否存在双轨状态
4. 弹窗读取的数据源是否为空或接错字段
5. 是否存在拆分后的旧逻辑残留

---

## 3. 运行中进度显示异常

### 固定排查顺序
必须优先按这条链查：

`日志原文 -> summary.progress -> /api/runs/active.progress -> 前端展示`

### 不要直接从这里逆推
- `preflight`
- 
除非明确是在处理：
- 完成态固化
- 历史兼容逻辑
- 缺失字段兜底

---

## 4. 总量 / 总数 / 百分比互相打架

### 典型现象
- 0% 时总量对，开始传输后总量变小，几秒后又恢复
- 总量正常了，但百分比仍然不对
- 已传数量和总数量不一致

### 优先排查
1. `progress.totalBytes` 是否被偏小 live 值覆盖
2. `plannedFiles` / `totalCount` 是否存在回退缺失
3. `percentage` 是否仍沿用旧语义计算
4. 接口输出时相关字段是否自洽
5. 前端是否做了 masking，掩盖了真实根因

### 已验证有效的经验
- 总量/总数问题优先修后端源头，不要默认先加前端显示保护
- 当总量语义被纠正后，百分比也要同步按真实总量重算

---

## 5. WebSocket 不刷新 / 卡片不自动更新

### 典型现象
- 后端已在推送，前端卡片不变
- 页面刷新后才看到最新状态

### 优先排查
1. 前端是否按正确字段匹配 active run
2. 当前结构是否应使用 `runRecord.id`
3. WebSocket 消息是否到达
4. 是否有轮询兜底
5. 当前页面是否实际消费了更新后的状态

### 已验证经验
- `run_progress` 更新 active run 时，必须按 `runRecord.id` 匹配

---

## 6. build 失败 / 样式异常 / 文件残片

### 典型现象
- `Invalid end tag`
- 样式截断
- 重复 `</style>`
- 某文件尾部出现莫名其妙残片

### 优先排查
1. 最近编辑文件是否存在尾部残留
2. 模板 / script / style 结构是否闭合
3. 是否有重复块
4. 是否在多轮编辑后留下旧内容

### 已验证经验
- 超大文件在频繁精确编辑时更容易出现尾部残片
- 这类问题常是文件编辑残留，不一定是业务逻辑本身

---

## 7. 本地构建结果和预期不符

### 优先判断
如果用户明确说明：
- 使用本地工作目录代码构建
- 使用 `docker build --no-cache ...`

则默认应先按“当前源码问题”排查，不应优先怀疑缓存或旧 bundle。

### 优先排查
1. 当前源码接线是否完整
2. 当前模板 / 脚本是否仍有旧引用
3. 当前构建产物是否与最新源码一致

---

## 8. 遇到复杂问题时的推荐处理顺序

1. 先收集现象
2. 先固定排查链
3. 先判断问题在哪一层
4. 先给出根因判断
5. 再决定修前端、后端还是两边都动
6. 改动尽量小步推进
7. 每步改完明确测试点

---

## 9. 一个简单原则

如果一个问题只能通过“前端再补一层显示保护”才看起来正常，通常说明你还没有真正修到根因。
