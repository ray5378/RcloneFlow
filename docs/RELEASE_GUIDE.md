# RELEASE_GUIDE.md

分支合并与发布收口指南。

这份文档用于约束从开发分支收口到主线时的基本流程，减少“代码能跑但主线状态不干净、规则没同步、产物带脏”的情况。

说明：这里的“默认配置项”指设置中的“默认配置”页，而不是代码里的常量默认值。

---

## 1. 当前分支原则

- `dev`：开发分支
- `master`：主线 / 稳定分支
- 正确流程：`dev -> master`
- 当前仓库只保留 `dev` / `master` 两个分支
- 不再使用 `next-master` 作为主线
- 不应执行 `master -> dev` 反向合并

### 当前发布状态（2026-04）

- 当前 `dev` 与 `master` 已对齐到同一提交
- 当前发布标签仅保留一个：`v1.0.0`
- 如需新发布，应在当前状态基础上更新版本与标签，而不是继续堆叠历史旧 tag

---

## 2. 什么时候适合从 `dev` 合到 `master`

满足以下条件时，通常可以评估合并：
- 这轮核心目标已经完成
- 关键用户可见回归已修复
- 关键验证已经通过
- 当前变更已经形成阶段性闭环
- 工作区是干净的，或已知脏改动已清理

---

## 3. 合并前检查

合并前至少确认：
- 当前改动目标已收口
- build / test 已通过到合理程度
- 关键页面已验证
- 文档已同步到位（如本轮形成了新规则或新阶段进度）
- 无关构建产物没有混入
- 无关临时代码、调试代码已清理

---

## 4. 合并时注意事项

- 优先保持一次合并承载一个阶段性主题
- 不要把大量无关杂项顺手一起混进主线
- 如果这轮包含“重构 + 回归修复 + 文档补充”，合并前应能说清这三者边界

---

## 5. 合并后检查

合并到 `master` 后至少确认：
- 当前分支正确
- 工作区干净
- 主线不存在明显未收口状态
- 文档索引与主文档能反映当前规则和阶段成果

---

## 6. 构建产物处理

当前项目中：
- `web/` 是前端构建输出目录
- 仓库只应保留可重建所需的源码与配置，不再把 `web/index.html` / `web/assets/*` 当作需要手工维护的受管源码

因此合并前应特别确认：
- 若本轮前端源码改动影响了打包结果，应通过重新构建验证，而不是手工维护构建产物
- 是否把与本轮无关的旧构建产物噪音误带入主线
- 本地构建加速缓存（如 `third_party/docker/`、`third_party/npm-cache/`、`third_party/go-mod-cache/`、`third_party/apk-cache/`）不应混入主线提交

如果本地直接启动 Go 服务时缺少 `web/`，应先执行 `cd frontend && npm run build`，或改用 Docker 镜像启动。

---

## 7. 什么时候不应该急着合并

以下情况通常应先停一下：
- 仍存在用户可见回归
- 关键链路语义还没理顺
- 当前改动范围太散，说不清这轮到底完成了什么
- 工作区仍混有无关脏改动
- 本轮重构刚做完，但尚未验证是否稳定

---

## 8. 默认配置项发布检查

如果本轮新增或修改了默认配置项，合并前至少确认：
- 默认值符合当前产品预期
- 设置页 / 默认配置页中可以看到该项
- 保存后后端已持久化
- 重新打开页面时可以正确回显
- 保存后主界面无需手动刷新即可按新值生效（如果该项设计为热生效）

例如当前已经落地的：
- `RUNNING_HINT_DEBUG_ENABLED`
  - 默认值：`false`
  - 作用：控制任务运行中提示小窗是否允许展开调试详情
  - 当前行为：默认关闭；开启后可展开自检结果、日志原文、接口进度 JSON

---

## 9. 文档收口要求

如果本轮改动带来了以下内容，应尽量在合并前补文档：
- 新规则
- 新约定
- 新排查顺序
- 新的拆分进度
- 新的阶段性结论

这样主线不仅有代码，也有能支撑后续维护的知识沉淀。

---

## 10. 一个简单原则

主线不是“先堆进去再说”的地方。

如果一轮改动还说不清：
- 改了什么
- 修了什么
- 还剩什么

那通常说明还不适合合到 `master`。
