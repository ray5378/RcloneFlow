# ERROR_HANDLING_GUIDE.md

错误处理指南。

这份文档用于统一前后端在错误处理上的基本思路，减少“错误被吞掉”“错误文案无意义”“错误链难定位”的问题。

---

## 1. 总目标

错误处理应优先做到：
- 可定位
- 可解释
- 不误导
- 不把主链问题藏起来

---

## 2. 后端错误处理原则

- 不要随意吞错
- 吞错必须有明确理由
- controller 负责把错误转成合理的 HTTP 响应
- service 应尽量保留足够语义，便于定位真实来源
- 不要让错误只剩下含糊的“失败了”

---

## 3. 前端错误处理原则

- 不要把所有错误都直接暴露成技术细节给最终用户
- 但也不要把所有错误都模糊成“出了点问题”
- 用户提示应尽量说明：
  - 发生了什么
  - 影响是什么
  - 还能不能继续

---

## 4. 调试与用户展示分离

- 原始报错、接口 JSON、调试详情应和用户主界面分离
- 默认用户界面应尽量展示可理解的结果
- 真正需要深挖时，再展开调试信息

---

## 5. 当前项目中的高风险错误处理区

- 运行中进度链
- 接口契约变化
- WebSocket 更新
- 弹窗接线
- 构建 / 模板残片 / 引用残留导致的前端运行时报错

这些区域的错误如果处理得太粗糙，后续排查会非常痛苦。

---

## 6. 一个简单原则

如果一个错误处理方式让问题“暂时不报了”，但后续没人知道为什么错、错在哪、还会不会再错，那通常不算好的错误处理。
