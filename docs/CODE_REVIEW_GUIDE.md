# CODE_REVIEW_GUIDE.md

代码审阅指南。

这份文档用于统一“怎么看改动是否靠谱”的标准，减少审阅时只看表面通过、不看结构风险的问题。

---

## 1. 审阅目标

代码审阅不只是看“能不能跑”，还要看：
- 结构是否更清楚
- 语义是否更一致
- 风险是否可控
- 是否引入新的技术债
- 是否容易在后续回归

---

## 2. 审阅时优先看什么

### 2.1 改动目标是否单一
先确认：
- 这轮改动主要是在修 bug、重构、文档、测试还是样式
- 是否无边界混进了无关改动

### 2.2 主数据源是否被破坏
重点检查：
- 有没有把单一真源又重新绕开
- 有没有把回退逻辑偷偷抬成主链
- 有没有把展示层 masking 重新引入主路径

### 2.3 旧接线是否清理干净
重点检查：
- 是否存在双轨实现
- 是否残留旧引用、旧状态、旧模板分支
- 是否出现“新代码已接上，但旧逻辑还留着”的情况

### 2.4 风险点是否已说明
对于高风险改动，应能从提交说明或上下文中看出：
- 这一步改了什么
- 影响了什么
- 应该测什么

---

## 3. 前端审阅重点

- 页面是否继续变得更大、更难理解
- 逻辑是否放在合理层级（view / component / composable / helper）
- 响应式状态是否出现隐蔽副作用链
- 模板是否堆了过多复杂表达式
- 弹窗、实时展示、状态绑定是否更容易回归
- 是否引入了新的“看似能显示、语义其实混乱”的补丁逻辑

---

## 4. 后端审阅重点

- 字段语义是否仍稳定
- 是否把不同语义的值混在一起返回
- 回退逻辑是否可解释
- 是否该用 typed struct 的地方还在堆 `map[string]any`
- controller 是否承担了过多业务规则
- 契约变化是否同步了测试与文档

---

## 5. 测试与验证审阅重点

- 是否补了最贴近根因的测试
- 是否只做了手工验证，没有固化关键回归
- build / test / 关键页面验证是否到位
- 是否把“应该测什么”说清了

---

## 6. 以下情况应提高警惕

- 一次提交同时改很多层，又说不清边界
- 提交说明过于模糊
- 代码里出现很多临时变量、临时回退、临时判断，但没有退出条件
- 通过前端修补掩盖后端语义问题
- 重构后旧代码没有清理
- 文档、测试、代码明显脱节

---

## 7. 一个简单原则

如果一段改动让后续维护者更难判断：
- 数据从哪来
- 为什么这样显示
- 真正该信哪个字段

那它通常值得被继续追问，而不是直接放过。
