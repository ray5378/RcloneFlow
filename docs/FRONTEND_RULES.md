# FRONTEND_RULES.md

前端开发规则。用于约束 Vue / TypeScript / 页面交互层在本项目中的实现方式，减少超大页面、状态混乱、回归频发的问题。

## 1. 总目标

前端代码应优先追求：
- 职责清晰
- 数据源明确
- 交互层可验证
- 页面结构可继续拆分
- 改动后不容易引入回归

不要只追求“先能显示出来”，而忽视结构与后续维护成本。

## 2. Vue 组件边界规则

### 2.1 页面组件（views）
页面组件应主要承担：
- 页面布局
- 区域装配
- 少量页面级编排逻辑
- 子组件之间的协调

页面组件不应长期承担：
- 大量格式化函数
- 复杂调试逻辑
- 多块可独立存在的 UI 区块
- 与多个数据链强耦合的细节逻辑

### 2.2 业务组件（components）
以下内容优先拆为组件：
- 独立弹窗
- 独立列表项区域
- 独立卡片区域
- 可被页面单独理解和维护的交互块

组件应优先保证：
- 输入清楚（props）
- 输出清楚（emits / 回调）
- 局部状态自洽

### 2.3 composable 边界
composable 适合承载：
- 可复用状态
- 派生数据
- 页面间复用的组合逻辑
- 与具体模板弱耦合的交互状态逻辑

composable 不应承载：
- 大块模板片段
- 强依赖具体 DOM 结构的逻辑
- 与单个页面模板细节完全绑定且无法复用的大量一次性逻辑

### 2.4 helper / 工具函数边界
helper 适合承载：
- 纯格式化
- 纯数据转换
- 无状态计算
- 可直接单测的纯函数逻辑

不要把需要读写大量响应式状态的逻辑伪装成 helper。

## 3. TypeScript 使用规则

### 3.1 优先明确类型
- 能写出明确类型时，不要默认使用 `any`
- 接口返回值、组件 props、关键状态对象、格式化输入输出，都应尽量显式标注类型

### 3.2 `any` 使用限制
以下情况禁止习惯性使用 `any`：
- 核心接口响应
- 关键进度字段
- 页面主状态对象
- 组件对外 props / emits

若短期必须使用 `any`：
- 只允许作为过渡
- 需要尽快收敛到更明确的类型
- 不要让 `any` 穿透整个调用链

### 3.3 命名语义
- 名称应体现真实含义，不要把 live / stable / preflight 等不同语义混成同名变量
- 布尔值命名应能直接回答“是否…”
- 计算结果、原始值、展示值尽量使用不同命名，避免混淆

## 4. 响应式状态规则

### 4.1 `ref` / `computed` / `watch` 使用原则
- `ref`：用于真正需要可变的响应式状态
- `computed`：用于稳定派生结果，不要把副作用塞进 `computed`
- `watch`：只在确实需要响应副作用时使用，不要用 `watch` 代替正常的数据建模

### 4.2 禁止事项
- 不要用层层 `watch` 拼出隐藏状态机
- 不要把本该由单一主数据源驱动的 UI，拆成多路 `watch` 手工同步
- 不要把“临时修显示”写成隐蔽的响应式副作用链

### 4.3 单一主状态原则
同一个 UI 区块，优先存在一个可解释的主状态来源。
不要长期维持：
- 一套接口值
- 一套本地拼接值
- 一套历史缓存值
- 一套模板即时修补值

并让它们在页面里并行竞争。

## 5. 模板与渲染规则

### 5.1 模板复杂度控制
- 模板中不要堆复杂表达式
- 复杂格式化、复杂条件判断、复杂拼接应提前抽到 `computed` / helper / composable

### 5.2 条件渲染
- `v-if` / `v-show` 的选择要有明确语义
- 不要为了临时修复，叠加多层条件渲染把模板变得不可理解

### 5.3 列表渲染
- `v-for` 必须优先使用稳定 key
- 不要用不稳定值或展示文本当 key
- 列表项状态应尽量与真实实体标识绑定

## 6. 组件通信规则

### 6.1 props / emits
- 父组件传入的 props 应尽量保持单向清晰
- 子组件对外行为应通过 emits 或清晰回调暴露
- 不要通过隐式读写父级状态制造耦合

### 6.2 `v-model` 使用
- 仅在双向绑定语义明确时使用 `v-model`
- 不要把复杂业务状态全部折叠进多个 `v-model`，让边界消失

## 7. 异步与副作用规则

### 7.1 请求与轮询
- 页面中的请求、轮询、定时器、WebSocket 订阅必须有明确的启动和清理逻辑
- 不要留下页面卸载后仍继续运行的定时器和订阅

### 7.2 UI 与接口边界
- 不要在模板层直接拼接接口细节
- 接口返回值若需要兼容、归一化、修正，应优先在更靠近数据层的地方做，而不是散落在多个 UI 点位

### 7.3 实时数据链
- 实时数据展示优先围绕主接口 / 主事件源组织
- 不要在多个点位各自补“抗噪”“修正”“缓存拼接”逻辑，导致后续无法判断谁是真值

## 8. 样式规则

### 8.1 样式边界
- 组件样式优先服务本组件
- 全局样式只放真正需要跨页面复用的基础规则
- 不要依赖偶然命中的全局类名实现组件关键视觉效果

### 8.2 相似视觉块
- 视觉相似不等于直接复制粘贴样式
- 如多个卡片区域有共同基底，应抽出公共样式语义，再局部扩展

## 9. 调试能力规则

- 调试信息必须与正常用户展示隔离
- 调试信息默认隐藏，可显式展开
- 不要把原始 JSON、原始日志、自检信息直接混入常规展示路径

## 10. 前端回归高风险区提醒

以下区域改动时默认高风险，应小步推进并优先验证：
- 实时进度展示
- 弹窗开关与内容绑定
- 列表项状态渲染
- WebSocket 数据接线
- 轮询与定时器
- 页面级组合状态

## 11. 前端测试建议

- 纯 helper 优先补单测
- composable 若承载关键状态逻辑，优先补单测
- 页面级回归至少保证 build 通过、关键页面不报错、主流程不断
- 用户可见显示 bug，优先把测试补在最接近根因的那一层，而不是只依赖手测

## 12. 一个简单判断标准

如果一段前端代码让你必须同时理解：
- 模板
- 接口返回
- 多个 watch
- 多套缓存
- 多个回退值
- 调试逻辑

才能知道它为什么显示成现在这样，那通常说明它已经该拆了。
