# AGENT_COLLABORATION_GUIDE.md

Agent 与不会写代码或不以写代码为主的开发者协作指南。

这份文档的目标，不是要求开发者先学会写代码再参与项目，而是建立一套高效协作方式，让：

- 需求表达更清楚
- 排查问题更快
- 测试更有效
- 改动风险更低
- 项目推进更持续
- 开发者在协作过程中逐步建立编程理解

---

## 1. 基本原则

### 1.1 开发者不需要先会写代码
开发者可以：
- 描述现象
- 描述预期
- 描述使用场景
- 做验证
- 做选择
- 决定优先级

Agent 负责：
- 结构化理解问题
- 定位代码位置
- 提出修复方案
- 执行修改
- 说明影响面
- 给出测试重点

不会写代码，不等于不能高效参与开发。

### 1.2 开发者负责业务与体验判断，Agent 负责技术落地
最理想的分工是：

开发者回答：
- 这个功能想达到什么效果
- 当前哪里不好用
- 哪种方案更符合实际使用习惯
- 哪个问题优先级更高

Agent 回答：
- 问题可能在哪一层
- 应该怎么拆分
- 风险在哪里
- 改哪几个文件更合适
- 这一步改完应该测什么

### 1.3 先把问题说清，再急着改代码
很多低效开发，不是因为代码难，而是因为问题本身没说清。

优先把以下三件事说清：
- 现象是什么
- 预期是什么
- 哪一步开始不对

---

## 2. 怎样提需求最有效

### 2.1 最有效的需求表达方式
尽量用下面这个结构：

1. **背景**：这是哪个页面 / 哪个功能 / 哪个使用场景
2. **现状**：现在是怎么表现的
3. **问题**：哪里不符合预期
4. **预期**：希望改成什么样
5. **限制**：有没有不希望动的地方

例如：

- 任务列表页面里，运行中卡片现在信息太乱
- 我希望先只优化显示结构，不动后端逻辑
- 不要一次大改，先拆出弹窗或进度区块

这种表达比“这里不好，改一下”有效很多。

### 2.2 如果说不清需求，也可以先说“痛点”
如果暂时说不出精确方案，也没关系。可以直接说：

- 哪里看着别扭
- 哪里用起来烦
- 哪一步最容易出错
- 哪个地方你每次都不敢动

这对 Agent 一样有价值。

### 2.3 指定“不动什么”能大幅提升效率
如果你明确知道这次不希望碰哪里，最好直接说出来，例如：

- 先不要动后端
- 先不要动数据库
- 先不要动小窗开关逻辑
- 先不要大拆，只修显示 bug

这会极大减少无效发散。

---

## 3. 怎样描述 bug 最有效

### 3.1 不需要讲代码，先讲现象
优先说：
- 什么时候发生
- 怎么复现
- 表现成什么样
- 正常应该是什么样

例如比起说：
- “这里逻辑错了”

更有效的是：
- “开始时 0% 显示是对的，传输一开始总量突然变小，过 8~10 秒又恢复”

这类描述非常适合 Agent 顺着链路定位根因。

### 3.2 最有价值的 bug 信息
如果能提供以下信息，排查效率会明显提高：

- 页面位置
- 操作步骤
- 出错时间点
- 控制台报错
- 接口返回
- 日志原文
- 截图或屏幕录像
- “刚刚还是好的，做了什么后开始不对”

### 3.3 现象描述优先于猜测
可以猜，但不要让猜测替代现象本身。

高效表达顺序应是：
1. 先描述事实
2. 再说自己的怀疑

例如：
- “前端显示 2 KB / 4 B，但日志里是 121 MiB / 335 MiB，我怀疑不是日志错，而是中间链路组装错了。”

这就非常高效。

---

## 4. 怎样和 Agent 高效协作开发

### 4.1 优先采用“小步开发 + 每步停测”
对于中高风险改动，最推荐的协作方式是：

1. 先说明这一步只改什么
2. 明确这一步会影响什么
3. 改完后立刻停下
4. 由开发者验证
5. 验证通过后再继续下一刀

这特别适合：
- 大页面拆分
- 实时展示逻辑
- 弹窗接线
- 状态绑定
- 接口链路修复

### 4.2 让 Agent 每一步都说清三件事
一个高质量开发回合，Agent 最好每次都说清：

- **这一步改什么**
- **这一步可能影响什么**
- **你现在重点测什么**

如果 Agent 没主动说清，可以直接要求：
- 先告诉我这一步的影响面
- 先告诉我测什么
- 先不要继续下一刀

### 4.3 复杂问题优先让 Agent 先定位根因，再决定怎么改
如果问题较复杂，不要急着让 Agent 直接改。

更好的顺序：
1. 先查链路
2. 先给出根因判断
3. 再决定修前端、后端、还是两边都动

这可以减少“先糊一层补丁，后面再返工”的情况。

## 5. 怎样高效配合测试

### 5.1 测试不是“随便看看”，而是按重点验证
每次改动后，最好按 Agent 给出的重点去测，不要无边界乱测。

例如这一步只修：
- 总量 / 总数 / 百分比一致性

那就优先测：
- 0% 时是否正确
- 开始传输后是否还会变小
- 后续是否稳定
- 三者是否一致

### 5.2 开发者测试时最有价值的反馈
测试反馈尽量不要只说：
- “不对”
- “有问题”
- “好了”

最好说成：
- 哪一项对了
- 哪一项还不对
- 哪一步开始不对
- 和之前相比是变好了还是变差了

例如：
- “总量和总数正常了，但进度百分比还是不对。”

这种反馈非常高效。

### 5.3 如果一次测不全，就优先测最关键链路
不用每次都把所有东西测遍。

更高效的做法是：
- 只测这一步最直接受影响的点
- 确认通过后再继续下一步

## 6. 如何让开发者逐步理解编程，而不是被代码劝退

### 6.1 不要求一次理解全部代码
高效学习不是上来先读完整个仓库，而是先建立：
- 页面
- 接口
- 数据链
- 状态流向

这些“能看懂一条主线”的理解。

### 6.2 优先理解这四件事
对于不会写代码的开发者，最值得先学会的不是语法，而是：

1. **这个功能的数据从哪里来**
2. **中间经过了哪些层**
3. **最后在哪里显示出来**
4. **哪里最容易在中间出错**

一旦掌握这个思路，你就已经不是“完全不懂代码”，而是开始能理解系统结构了。

### 6.3 可以主动要求 Agent 用“链路图思维”解释
如果看不懂代码，可以让 Agent 用这种方式解释：

- 输入是什么
- 中间经过哪几层
- 输出是什么
- 哪一层最可能出错

这比直接贴函数代码更容易建立理解。

### 6.4 先学“判断问题在哪一层”，再学具体语法
真正提升效率的，不是立刻学会写函数，而是慢慢学会判断：

- 这是前端展示问题
- 这是接口组装问题
- 这是状态回退问题
- 这是日志解析问题

这类判断能力，会大幅提升与 Agent 协作效率。

---

## 7. 推荐协作句式

以下句式通常很高效。

### 7.1 提需求时
- 这次只想先解决……
- 先不要动……
- 我希望按小步改，每步停下来测试
- 你先告诉我这一步会影响什么
- 你先不要改，先查根因

### 7.2 提 bug 时
- 复现步骤是……
- 正常应该是……
- 现在实际是……
- 从……这一步开始不对
- 日志里是……，前端里是……

### 7.3 测试反馈时
- 这项正常了，但……还不对
- A 正常，B 不正常，C 还没测
- 比之前好了，但还有一个问题是……
- 这一步没问题，可以继续下一刀

### 7.4 学习理解时
- 你用链路方式给我讲一下
- 你告诉我这次是前端问题还是后端问题
- 你告诉我这几个字段分别是什么意思
- 你告诉我为什么会出现这个现象

---

## 8. 哪些协作方式低效，应尽量避免

以下方式容易显著拉低效率：

- 一次同时提很多互相无关的问题
- 没描述现象，就直接要求“大改”
- 回归出现后还要求继续推进下一刀
- 不区分“现象”“猜测”“结论”
- 测试后只说“还是不行”，不补充哪里不行
- Agent 已经说明影响面和测试点，但验证时完全不按重点测

---

## 9. 最终目标

高效协作的目标不是让开发者变成代码机器，也不是让 Agent 变成只会执行命令的工具。

而是形成一种更高效的合作方式：

- 开发者越来越会描述问题
- Agent 越来越会结构化落地
- 每次改动风险越来越低
- 项目推进越来越稳
- 开发者对系统和编程的理解越来越强

如果协作方式是对的，即使开发者本身不会写代码，也一样能持续把项目推进下去，并且在过程中逐步建立真正有用的编程理解。
