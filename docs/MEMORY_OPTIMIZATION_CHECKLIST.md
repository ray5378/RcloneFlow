# RcloneFlow 内存回收与占用优化清单

> 目标：先把现状、风险点、优先级和改造步骤固定下来，再按低风险高收益顺序逐步实施。

## 一、当前结论

当前前后端整体上 **没有看到必然失控的硬泄漏**，但也还不够“内存友好”。

### 当前优点
- 前端主要轮询定时器基本都有 `onUnmounted` 清理。
- Active Transfer 详情关闭时会主动清空大部分弹窗状态。
- 后端运行结束后会调用 `activeMgr.RemoveState(run.ID)` 释放运行态传输状态。
- 后端已有历史记录清理、日志清理服务。

### 当前主要问题
- 前端 WebSocket 生命周期管理不够干净，存在重复订阅/重复初始化风险。
- 前端 Run 详情文件列表采用“前端全量拉取 + 本地过滤分页”，大任务时内存压力偏高。
- 前端任务页/历史页偏高频轮询，WebSocket 与轮询有重复兜底，存在额外 CPU/内存抖动。
- 后端 Active Transfer 在运行期间会把全量 `Completed/Pending/Candidates` 常驻内存。
- 后端 Active Transfer 的 snapshot 持久化 + WebSocket 广播频率可能过高。
- 后端 WebSocket Hub 的 client 移除逻辑并发实现不够稳。

---

## 二、优化总原则

1. **先减小高频对象创建与序列化频率**
   - 优先削减无意义的全量广播、全量拉取、重复轮询。

2. **先做低风险高收益改造**
   - 不先碰业务语义复杂的核心传输逻辑。
   - 先做分页、节流、订阅生命周期整理。

3. **大数据场景优先考虑降级策略**
   - 当文件量很大时，不必强求全量明细全驻内存。

4. **每一步都要可验证**
   - 改完后至少做：构建、关键交互验证、必要时加埋点/统计。

## 四、详细执行清单

## 任务 1：前端 Run 详情改为真正服务端分页/筛选

### 问题
当前 `useRunDetailFiles.ts` 会循环把接口所有页全部拉回来，存入 `runFiles.value`，然后再在前端：
- filter
- paginate
- slice

文件量大时会导致：
- 浏览器内存上涨
- computed/filter/slice 反复处理大数组
- 打开详情卡顿

### 目标
改成：
- 前端只请求当前页
- 后端支持分页参数
- 如有需要，后端支持状态筛选（all/success/failed/other）

### 建议步骤
- [x] 梳理当前 run files API 能力
- [x] 设计后端分页/筛选返回结构
- [x] 改造前端 `useRunDetailFiles.ts` 为按页请求
- [x] 保留现有 UI 行为，但去掉前端全量缓存
- [x] 构建验证 + 手动验证详情翻页

### 预期收益
- 明显降低前端内存占用
- 减少大数组带来的 GC 压力
- 大历史任务详情更流畅

---

## 任务 2：后端 Active Transfer snapshot 持久化/广播节流

### 问题
当前 Active Transfer 状态更新时，会频繁触发：
- `db.UpdateRun(...)`
- `websocket.Broadcast("active_transfer_snapshot", ...)`

如果进度更新很密集，会导致：
- JSON 序列化频繁
- DB 写入频繁
- WebSocket 消息频繁
- 前后端都产生不必要的对象分配和内存抖动

### 目标
为每个 run 的 snapshot 更新增加节流/合并机制：
- 普通进度更新按时间窗口合并（如 200ms~500ms）
- 任务完成/失败/关键状态切换时立即 flush

### 建议步骤
- [x] 找到 snapshot persist/broadcast 的统一出口
- [x] 设计节流策略（按 runID）
- [x] 实现节流队列/最后一帧覆盖
- [x] 对完成态/失败态保留立即发送
- [x] 压测或最少做日志验证，确认广播量下降

### 预期收益
- 降低后端 CPU 与堆抖动
- 降低 DB 写压力
- 降低前端消息处理压力

---

## 任务 3：前端 WebSocket 生命周期收口

### 问题
当前 `useWebSocket.ts` 使用全局单例状态，但又在组件生命周期里 connect/disconnect；
`setupRealtimeSync()` 也有重复注册监听的风险。

### 风险
- 重复订阅
- 同一消息触发多次
- 页面切换时全局 socket 被意外断开
- 多余 reconnect 与回调残留

### 目标
统一 WebSocket 使用模型：
- 要么彻底单例化
- 要么彻底实例化

推荐方向：**全局单例连接 + 组件仅订阅/退订消息**

### 建议步骤
- [x] 梳理当前 `useWebSocket` / `onWsMessage` 的调用点
- [x] 给 `setupRealtimeSync()` 增加幂等保护
- [x] 为订阅返回值统一做 `unsubscribe` 清理
- [x] 组件卸载时不再随意关闭全局 socket
- [x] 验证任务页切换时不会重复收消息

### 预期收益
- 降低前端长期运行后的事件残留风险
- 减少无意义刷新与内存抖动

---

## 任务 4：任务页/历史页轮询策略动态化

### 问题
当前部分页面固定 3 秒轮询，虽然有可见性判断，但整体仍偏粗。

### 目标
让轮询频率根据状态动态调整：
- 有活跃任务：高频（如 3s）
- 无活跃任务：低频（如 10~30s）
- 历史静态详情：更低频或仅手动刷新

### 建议步骤
- [x] 梳理 `loadActiveRuns` / `refreshTaskHistoryRuns` 的触发路径
- [x] 提取统一轮询策略配置
- [x] 对“无运行任务”场景降频
- [x] 尽量让 WebSocket 做主、轮询做兜底
- [x] 验证切模块/切页面时定时器数量不增长

### 预期收益
- 降低前端请求量
- 降低对象创建频率
- 减少不必要的内存与 CPU 消耗

---

## 任务 5：后端 Active Transfer 大任务降级策略

### 问题
当前运行中会把：
- `Candidates`
- `Pending`
- `Completed`
- `CurrentFiles`

整场任务常驻内存直到结束。任务很大时，heap 会明显上涨。

### 目标
在大文件量场景下，提供 lightweight tracking：
- 保留 summary / current files / counters
- 不强求全量 completed/pending 全驻内存
- 或仅保留最近 N 条 completed

### 建议步骤
- [x] 定义降级触发阈值（如候选数 > N）
- [x] 明确降级后哪些 API 字段仍保留
- [x] 实现内存态裁剪策略
- [x] 前端 UI 对 degraded 模式友好展示
- [x] 验证大任务下内存曲线

### 预期收益
- 控制大任务运行时的后端内存峰值
- 避免 completed/pending map 无限膨胀到任务结束

---

### 目标
仅缓存必要字段，限制体积和写入频率。

### 建议步骤
- [x] 统计 tasks 快照字段规模
- [x] 设计最小缓存结构
- [x] 限制写入频率
- [x] 增加版本号和异常回退

## 任务 7：WebSocket Hub 并发整理

### 问题
`hub.go` 中广播阶段对 client 的移除逻辑写法不够稳，读锁下修改 map 存在隐患。

### 目标
重构为更稳的 client 生命周期管理：
- 广播阶段只收集坏 client
- 统一在写锁阶段移除

### 建议步骤
- [x] 梳理当前 register/unregister/broadcast 流程
- [x] 修正广播时 client 清退逻辑
- [x] 验证断开连接与异常写失败场景

## 八、本轮完成摘要

本轮已完成清单中的 1-7 项核心优化，重点结果如下：

- Run 详情已从“前端全量拉取 + 本地过滤分页”改为真正的服务端分页/筛选，降低大历史任务详情的浏览器内存压力。

- 前端 active runs / history 列表已补最小更新策略，减少整包替换带来的响应式重算与重渲染。
- 后端 Active Transfer 已支持大任务 degraded 模式：保留真实计数，同时只保留最近一部分 completed/pending 明细，控制运行时内存峰值。
- `lastTasksSnapshot` 已做最小字段缓存、版本化包装和节流写入，减少 localStorage 体积与前端序列化开销。
- WebSocket Hub 广播清退逻辑已改为“读锁收集、写锁统一移除”，避免读锁阶段修改 client map 的并发隐患。

验证结果：
- 前端 `npm run build` 通过
- 后端 `go test ./internal/active_transfer/... ./internal/controller/... ./internal/websocket/...` 通过
