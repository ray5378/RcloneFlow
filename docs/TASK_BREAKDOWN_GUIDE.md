# TASK_BREAKDOWN_GUIDE.md

任务拆解指南。

这份文档用于指导如何把一个大需求拆成可验证、可推进、可回归控制的小任务。

---

## 1. 拆解目标

拆任务不是为了把事项写得更碎，而是为了：
- 降低改单步风险
- 让验证更聚焦
- 让回归更容易定位
- 让开发者更容易判断当前进度

---

## 2. 推荐拆解方式

优先按以下维度拆：
- 数据链
- UI 区块
- 状态管理
- 接口契约
- 风险层级

例如不要直接写：
- “重构整个页面”

而要拆成：
- 拆 helper
- 拆 composable
- 拆弹窗
- 修接线
- 做回归验证

---

## 3. 好任务的特征

一个拆得比较好的任务，通常应满足：
- 范围清楚
- 目标明确
- 改完能立即验证
- 风险相对可控
- 和其他任务的边界清楚

---

## 4. 一个简单原则

如果一个任务做完后，别人还说不清“这一步到底完成了什么”，那它通常拆得还不够好。
